# CLAUDE.md — 기본 지침서

이 문서는 모든 프로젝트에 공통 적용되는 개발 대원칙이다. 코드를 분석, 작성, 수정하는 모든 과정에서 반드시 준수한다.

---

## 1. 언어 원칙

- **모든 응답, 사고 과정, 설명은 반드시 한국어로 작성**한다.
- 코드 내 변수명/함수명은 영어, 주석과 UI 텍스트는 한국어를 사용한다.
- 커밋 메시지와 PR 설명도 한국어로 작성한다.

## 2. 코드 작성 원칙

### 스타일 가이드
- Google Style Guide를 기본으로 따른다.
- ESLint, Prettier 설정을 프로젝트에 포함하고 일관된 포맷을 유지한다.
- TypeScript strict 모드를 사용한다. `any` 타입 사용을 금지한다.

### 모듈화
- 모든 기능은 적절한 단위로 분리하여 모듈화한다.
- 각 모듈에 명확한 이름을 부여하고, 단일 책임 원칙(SRP)을 준수한다.
- 특정 기능 수정 시 다른 모듈에 영향이 없도록 독립성을 확보한다.
- RESTful API 설계 원칙을 따르고, 내부 API를 적극 활용하여 모듈 간 결합도를 낮춘다.

### 최소 구현
- **요청받은 내용에 대해서만 보수적으로 구현**한다.
- 임의로 관련 기능을 추가하거나 확장하지 않는다.
- 불필요한 추상화, 미래 대비 설계, 과도한 일반화를 하지 않는다.
- 특정 필드/케이스 전용 로직을 다른 필드에 적용하지 않는다.

### 파일/토큰 절약
- 설명용 .md 파일을 불필요하게 생성하지 않는다.
- .md 파일에 과도한 예시 코드를 넣지 않는다.
- 기존 파일 수정을 새 파일 생성보다 우선한다.

## 3. 트러블슈팅 원칙

- 오류 수정 시 **근본 원인을 반드시 파악**한다.
- 임시 우회나 핫픽스가 아닌 근본적 해결을 한다.
- 같은 오류 재발 방지를 위한 지침을 이 문서에 업데이트한다.
- 디버깅 시 에러 메시지를 정확히 읽고, 스택 트레이스를 추적한다.

## 4. Git 원칙

- 커밋은 의미 있는 단위로 나눈다.
- 커밋 메시지 형식: `[타입] 간단한 설명` (feat, fix, refactor, docs, style, test)
- 작업 브랜치에서 개발 후 PR을 통해 병합한다.

## 5. 보안

- 환경변수(.env)를 코드에 하드코딩하지 않는다.
- .env 파일을 .gitignore에 반드시 포함한다.
- 클라이언트에 노출되는 키는 읽기 전용/공개 키만 사용한다.
